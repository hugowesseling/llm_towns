# LLM Towns Architecture

## Overview

LLM Towns is a monolithic Python simulation of a medieval fantasy world where AI-driven villagers autonomously pursue goals generated by Large Language Models (LLMs). The system combines deterministic simulation mechanics with AI-driven decision-making to create emergent village behavior, including inter-town travel and trade.

The application runs as a Flask web server providing REST API endpoints for world inspection and simulation control, while a background thread continuously advances the simulation state. Villagers navigate a 2D grid world, trade goods at markets, form relationships, and can travel between towns — all through LLM-generated goals and plans.

## World Grid & Town Tiles

The world is a **single flat 2D grid** (Option B). Towns are not on separate sub-grids; instead, each town's bounding box is filled with specialized tiles directly on the main grid:

| Tile | ID | Walkable | Purpose |
|------|----|----------|---------|
| `GRASS` | 0 | Yes | Default terrain |
| `FOREST` | 1 | Yes | Wooded areas |
| `WATER` | 2 | No | Rivers, lakes |
| `MOUNTAIN` | 3 | No | Impassable terrain |
| `SAND` | 4 | Yes | Desert biome |
| `SNOW` | 5 | Yes | Tundra biome |
| `SWAMP` | 6 | Yes | Swamp biome |
| `ROAD` | 7 | Yes | World roads + town streets |
| `BUILDING` | 8 | No | Shops, temples, guilds (town perimeter) |
| `HOUSE` | 9 | No | Residential (town interior) |
| `TOWN_SQUARE` | 10 | Yes | Central plaza (3×3) |

Each town has **4 entry/exit points** (road tiles at the midpoint of each bounding-box side). Roads between towns connect via these entry points. Villagers A*-path around buildings via the street network.

## Core Behavior

The simulation operates on a tick-based system where each second advances one simulation tick. During each tick:

1. **Goal Assignment**: Idle villagers periodically receive new goals from the LLM based on their current state (needs, inventory, position, relationships) **plus town context** (home town name, nearby towns, inter-town relations)
2. **Plan Generation**: Goals are converted into executable action sequences, either via LLM planning or deterministic fallbacks. Supported action types now include **`travel`** for inter-town movement.
3. **Action Execution**: Active actions progress through states (ready → running → completed), with A* pathfinding for movement and collision detection. Walk destinations resolve to town-aware positions (correct market per town, town center for "home"/"square").
4. **Trading**: Villagers buy/sell goods at their **home town's market** (no longer hardcoded to `market_1`).
5. **Town Context in LLM**: Every goal and plan prompt includes `build_town_context()` — a string listing the villager's home town and all other towns with their diplomatic relations.
6. **Travel**: The `travel` action pathfinds from the villager's current position to the target town's south entry point along the road network.
7. **Town Name Propagation**: Interaction memories include town affiliations (`"Talked with Elara from Stonebrook"`), so town names spread through the social graph.

The system gracefully degrades when LLM services are unavailable, falling back to predefined behaviors while maintaining simulation continuity.

## Key New Features (v2)

- **Town tile grid** — towns rendered as building/house/square tiles on the main world grid
- **Entry/exit points** — 4 cardinal gates per town, stored in `Town.entry_points`
- **Town-aware markets** — each town has its own market, resolved by `villager.town`
- **Travel action** — inter-town movement via A* along roads
- **Town context in prompts** — `build_town_context()` surfaces town names and relations to the LLM
- **Town names in memories** — interaction records include `"from Stonebrook"` style references

## Module Architecture

### app.py - Main Application Server

**Purpose**: Flask web server providing REST API endpoints and simulation bootstrap.

**Key Behaviors**:
- Generates world with town tile grids, entry/exit points, and road networks
- Starts background simulation thread running at 1 tick/second
- Provides endpoints for world/town/character inspection (town data includes `entry_points`)
- Grid legend includes new tile types (8=building, 9=house, 10=town_square)
- Interaction endpoint appends town names to memories (`"Talked with Elara from Stonebrook"`)
- Possible-actions endpoint exposes road-connected towns as travel targets
- Handles goal assignment and LLM suggestions
- Manages static character data alongside dynamic simulation state

**Interactions**:
- Imports and initializes `SimulationScheduler` with planner and LLM brain
- Calls scheduler methods for goal assignment and status snapshots
- Serves as the primary interface between external clients and simulation core
- Villager IDs follow `{town_id}_villager_{n}` format (e.g., `town_0_villager_3`)

### simulation/scheduler.py - Simulation Core

**Purpose**: Central coordinator managing all simulation entities and the main tick loop.

**Key Behaviors**:
- Maintains collections of villagers, goals, plans, markets, and world grid
- Executes the main `advance_tick()` loop calling `_tick_villagers()`
- Performs A* pathfinding on the world grid for villager movement
- Enforces collision detection preventing overlapping positions
- **Executes `walk` actions** — resolves named destinations (`market`, `home`, `square`) to town-aware positions (e.g., each villager walks to their own town's market)
- **Executes `travel` actions** — pathfinds to the target town's south entry point along roads
- **Executes `trade` actions** — resolves the correct market by villager's home town (`_get_market_for_villager`)
- Automatically assigns new goals to idle villagers using LLM with **town context** injected into prompts
- Provides `move_villager()` for position updates with collision + walkability checks

**Interactions**:
- Uses `Planner` to generate plans from goals
- Calls `LLMBrain` for goal suggestions when villagers are idle
- Builds `town_context` via `build_town_context()` for LLM prompts
- Updates villager positions and inventories during action execution
- Logs events (goal added, plan completed, trades) to event log

### simulation/planner.py - Goal-to-Plan Conversion

**Purpose**: Translates high-level goals into executable action sequences.

**Key Behaviors**:
- Attempts LLM-based plan generation with structured JSON prompts — includes **town context** from `build_town_context()`
- Falls back to deterministic plan templates when LLM unavailable
- Parses LLM responses into `Action` objects with durations and metadata
- Supports `travel` action type in default durations (30 ticks) and action building
- Fallback planner detects town names in goal descriptions and generates `travel` actions
- Builds action sequences for common goals (food acquisition, socialization, inter-town visits)

**Interactions**:
- Receives `Villager` and `Goal` objects from scheduler
- Uses `LLMBrain.create_chat_json()` for plan generation
- Returns `Plan` objects containing queued actions
- Accesses scheduler for context (world towns, markets, positions) during plan creation

### simulation/villager.py - Villager Entity Model

**Purpose**: Data model representing individual villagers with state and behaviors.

**Key Behaviors**:
- Tracks position, inventory, needs (hunger/energy/social), relationships
- Provides `summary()` method for LLM context generation
- Manages goal/plan assignment and idle state detection
- Handles inventory operations (add/remove items)

**Interactions**:
- Used by scheduler for state updates and collision checking
- Passed to planner for goal/plan generation context
- Updated by market operations during trading
- Referenced by actions for actor identification

### simulation/actions.py - Goal-Plan-Action Hierarchy

**Purpose**: Hierarchical action system with state management and execution lifecycle.

**Key Behaviors**:
- Defines `Action` with states (planned → ready → running → completed/failed)
- Implements `Goal` (high-level objectives) and `Plan` (action sequences)
- Manages action progression through ticks with duration tracking
- Supports action interruption and subaction hierarchies

**Interactions**:
- Actions are created by planner and executed by scheduler
- Plans contain action queues managed by scheduler tick loop
- Goals are assigned by scheduler and converted to plans by planner

### simulation/decision.py - Cognitive Decision Timing

**Purpose**: Layered cognition system controlling decision-making frequency.

**Key Behaviors**:
- Defines three decision layers: reactive (1 tick), tactical (30 ticks), reflective (300 ticks)
- Tracks last execution tick for each layer per villager
- Determines when villagers should evaluate new goals

**Interactions**:
- Used by scheduler to decide when to assign new goals
- Integrated into `VillagerCognition` dataclass attached to each villager

### simulation/market.py - Trading System

**Purpose**: Manages market-based and peer-to-peer trading mechanics.

**Key Behaviors**:
- Handles buy/sell operations with price calculations and inventory transfers
- Supports direct villager-to-villager trading
- Maintains market inventory and pricing data

**Interactions**:
- Called by scheduler during trade action execution
- Updates villager inventories during successful transactions
- Position-checked by scheduler for proximity-based trading

### llm/brain.py - LLM Integration Layer

**Purpose**: Abstraction layer for LLM API interactions with structured response handling.

**Key Behaviors**:
- Wraps OpenAI-compatible chat completion APIs
- Provides `chat_json()` methods for structured JSON responses
- Handles API errors and response parsing
- Supports both simple chat and structured JSON modes

**Interactions**:
- Used by planner for plan generation
- Used by scheduler for goal suggestions
- Receives prompt messages from `prompts.py` builders

### world/world_generator.py - World Generation

**Purpose**: LLM-driven world creation system for generating terrain, towns, and initial world state.

**Key Behaviors**:
- Uses LLM to generate world metadata (name, description, climate, era)
- Procedurally places terrain features (forests, mountains, rivers) on world grid
- Generates towns with buildings, population, economy, and culture
- **Generates town tile layout** on the main world grid via `_generate_town_layout()`:
  - Clears town area to grass
  - Draws N-S and E-W cross streets as `ROAD` tiles
  - Places 3×3 `TOWN_SQUARE` at center
  - Fills perimeter with `BUILDING` tiles, interior with `HOUSE` tiles
  - Sets 4 entry/exit points (`entry_points` dict: north/south/east/west)
- Roads are traced between **town centers** (not top-left corners), naturally passing through entry points
- Creates initial villager populations — **spawns on walkable tiles** (roads/square) within each town
- Updates world grid with terrain obstacles and walkable areas
- New terrain tile types: `BUILDING` (8, non-walkable), `HOUSE` (9, non-walkable), `TOWN_SQUARE` (10, walkable)

**Interactions**:
- Called by app.py during world initialization instead of hardcoded world data
- Uses `LLMBrain` for creative world description generation
- Provides `World` object to scheduler for pathfinding and collision detection
- Generates `Town` data (including `entry_points`) consumed by simulation and API
